# PRD: AI 自动生成任务描述

## 1. 功能概述

当用户创建新任务时仅输入简短标题，系统通过 RAG（检索增强生成）技术自动生成结构化的任务描述，包括详细描述、验收标准和技术方案建议。

## 2. 业务价值

- **提升效率**：减少用户手动编写任务描述的时间，从平均 5-10 分钟降至 30 秒
- **标准化**：确保任务描述格式统一，包含必要的关键信息
- **知识复用**：利用项目历史数据，传承团队最佳实践
- **降低门槛**：帮助新成员快速创建高质量任务

## 3. 用户场景

**主场景**：
1. 用户在任务看板点击"创建任务"
2. 输入简短标题（如"用户登录优化"）
3. 点击"AI 生成描述"按钮
4. 系统在 2-3 秒内返回结构化描述建议
5. 用户可直接使用或编辑后保存

**典型输入示例**：
- "修复支付接口超时问题"
- "添加数据导出功能"
- "优化首页加载速度"

## 4. 功能需求

### 4.1 核心功能

#### 4.1.1 RAG 检索流程
1. **语义检索**：根据任务标题检索项目历史中相似的任务
   - 使用向量相似度匹配（Embedding）
   - 检索范围：当前项目的所有已完成任务
   - 返回 Top 5 最相似任务

2. **上下文提取**：从检索结果中提取关键信息
   - 任务描述
   - 验收标准（Acceptance Criteria）
   - 技术方案/实现细节
   - 关联的标签和优先级

3. **生成结构化描述**：调用 LLM 生成任务描述
   - 输入：用户标题 + 检索到的历史任务上下文
   - 输出：结构化 Markdown 格式描述

#### 4.1.2 生成内容结构

```markdown
## 任务描述
[详细的功能描述，说明要做什么、为什么做]

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
- [ ] 标准 3

## 技术方案建议
- 前端：[具体技术栈和实现思路]
- 后端：[API 设计和数据处理]
- 数据库：[表结构变更或查询优化]

## 参考任务
- #123 [相似任务标题]
- #456 [相似任务标题]
```

### 4.2 UI 交互

#### 4.2.1 创建任务弹窗增强
- 在"描述"字段上方添加"✨ AI 生成描述"按钮
- 按钮状态：
  - 默认：可点击（蓝色）
  - 生成中：禁用 + Loading 动画
  - 生成完成：自动填充到描述框
  - 生成失败：显示错误提示

#### 4.2.2 生成结果展示
- 描述框自动填充生成内容
- 支持 Markdown 预览和编辑模式切换
- 显示"参考任务"链接，点击可查看原任务详情

### 4.3 非功能需求

- **响应时间**：95% 请求在 3 秒内返回
- **准确性**：生成内容与用户意图匹配度 > 80%（通过用户反馈评估）
- **可用性**：服务可用率 > 99%
- **并发**：支持 50 QPS

## 5. 技术方案

### 5.1 架构设计

```
Frontend (React)
    ↓ HTTP POST /api/v1/tasks/ai/generate-description
Backend (Spring Boot)
    ↓ 1. 调用 Embedding API 生成标题向量
    ↓ 2. 向量检索相似任务（PostgreSQL pgvector 或 Redis）
    ↓ 3. 构造 Prompt + 调用 LLM API
    ↓ 4. 返回生成结果
Frontend
    ↓ 填充到描述框
```

### 5.2 技术栈

#### 后端
- **向量数据库**：PostgreSQL + pgvector 扩展（或 Redis Vector Search）
- **Embedding 模型**：OpenAI text-embedding-3-small 或开源模型（如 BGE）
- **LLM**：Claude 3.5 Sonnet / GPT-4 / 开源模型（如 Qwen）
- **缓存**：Redis 缓存生成结果（相同标题 24 小时内复用）

#### 前端
- **Markdown 编辑器**：react-markdown + react-simplemde-editor
- **状态管理**：Zustand 管理生成状态

### 5.3 数据模型

#### 5.3.1 任务向量存储（新表）
```sql
CREATE TABLE task_embeddings (
    id BIGSERIAL PRIMARY KEY,
    task_id BIGINT NOT NULL REFERENCES tasks(id),
    title_embedding VECTOR(1536),  -- 向量维度取决于模型
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    UNIQUE(task_id)
);

CREATE INDEX idx_task_embeddings_vector ON task_embeddings
USING ivfflat (title_embedding vector_cosine_ops);
```

#### 5.3.2 AI 生成日志（新表）
```sql
CREATE TABLE ai_task_generation_logs (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES users(id),
    project_id BIGINT NOT NULL REFERENCES projects(id),
    input_title VARCHAR(500) NOT NULL,
    generated_description TEXT,
    reference_task_ids BIGINT[],  -- 参考的任务 ID 数组
    model_name VARCHAR(100),
    tokens_used INTEGER,
    response_time_ms INTEGER,
    is_accepted BOOLEAN,  -- 用户是否采纳
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 5.5 核心流程代码（伪代码）

```java
@Service
public class TaskAiService {

    public GenerateDescriptionResponse generateDescription(
        Long projectId, String title) {

        // 1. 生成标题向量
        float[] embedding = embeddingService.embed(title);

        // 2. 向量检索相似任务
        List<Task> similarTasks = taskRepository
            .findSimilarByEmbedding(projectId, embedding, 5);

        // 3. 构造 Prompt
        String prompt = buildPrompt(title, similarTasks);

        // 4. 调用 LLM
        String description = llmClient.generate(prompt);

        // 5. 记录日志
        logGeneration(projectId, title, description, similarTasks);

        return new GenerateDescriptionResponse(description, similarTasks);
    }

    private String buildPrompt(String title, List<Task> similarTasks) {
        return """
            你是一个敏捷项目管理助手。用户创建了一个新任务，标题是："%s"

            以下是项目历史中相似的任务：
            %s

            请生成结构化的任务描述，包括：
            1. 任务描述（详细说明功能需求）
            2. 验收标准（3-5 条可验证的标准）
            3. 技术方案建议（前后端实现思路）

            使用 Markdown 格式输出。
            """.formatted(title, formatSimilarTasks(similarTasks));
    }
}
```

## 7. 成功指标

- **使用率**：30% 的新任务使用 AI 生成描述
- **采纳率**：70% 的生成结果被用户直接采纳或轻度编辑后使用
- **时间节省**：平均任务创建时间减少 60%
- **用户满意度**：NPS > 8 分

## 8. 风险和缓解

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM API 不稳定 | 功能不可用 | 实现降级策略，返回简单模板 |
| 生成内容质量差 | 用户不信任 | 持续优化 Prompt，收集反馈迭代 |
| 向量检索性能差 | 响应慢 | 使用专业向量数据库，优化索引 |
| 成本过高 | 预算超支 | 设置每日调用上限，缓存结果 |

## 9. 验收标准

### 9.1 功能验收
- [ ] 用户在创建任务弹窗中输入标题后，可见"AI 生成描述"按钮
- [ ] 点击按钮后，按钮显示 Loading 状态且不可再次点击
- [ ] 3 秒内返回生成结果并自动填充到描述框
- [ ] 生成的描述包含"任务描述"、"验收标准"、"技术方案建议"三个部分
- [ ] 描述框支持 Markdown 预览和编辑模式切换
- [ ] 显示参考任务列表，点击可跳转到对应任务详情页

### 9.2 性能验收
- [ ] API 响应时间 P95 < 3 秒
- [ ] 支持 50 并发请求无错误
- [ ] 相同标题 24 小时内命中缓存，响应时间 < 500ms

### 9.3 质量验收
- [ ] 生成的描述与任务标题语义相关
- [ ] 验收标准至少包含 3 条可验证的条目
- [ ] 技术方案建议覆盖前端、后端、数据库三个层面
- [ ] 参考任务相似度 > 0.7（通过人工抽样验证）

### 9.4 异常处理验收
- [ ] LLM API 调用失败时，显示友好错误提示
- [ ] 项目无历史任务时，仍能生成基础描述模板
- [ ] 网络超时时，前端显示超时提示并允许重试
- [ ] 生成失败时，不影响用户手动输入描述

### 9.5 数据验收
- [ ] 每次生成记录到 `ai_task_generation_logs` 表
- [ ] 日志包含用户 ID、项目 ID、输入标题、生成结果、响应时间
- [ ] 用户保存任务时，更新日志的 `is_accepted` 字段

## 10. 未来扩展
