# 后端监控功能需求文档（Java 程序实时监控）

## 1. 文档目的

为当前系统新增一套后端运行监控能力，对 Java 服务的关键指标进行实时采集、展示与告警，覆盖 CPU、内存、JVM、访问量等核心维度，提升问题定位效率，支撑后续调试与稳定性优化。

## 2. 背景与问题

当前系统在运行态可观测性方面存在不足：缺少统一指标面板、JVM 深度指标、接口访问与慢请求监控、告警闭环。

## 3. 建设目标

1. 实时监控后端 Java 服务关键运行指标。
2. 对异常波动进行阈值告警（高 CPU、高内存、Full GC、慢请求）。
3. 提供可视化仪表盘，支持按时间窗口分析趋势。
4. 支持开发/测试环境快速定位问题，后续可平滑扩展到生产。

## 4. 监控范围

### 4.1 主机与进程资源
- CPU 使用率（系统/进程）
- 内存使用率（系统内存、进程驻留内存）
- 磁盘使用率（可选）
- 网络流量（可选）

### 4.2 JVM 指标
- Heap 使用量（used/committed/max）
- Non-Heap 使用量
- GC 次数与耗时（按收集器分类）
- 线程数（总线程、守护线程、峰值线程）
- 类加载数（已加载/卸载）

### 4.3 应用访问与性能指标
- 请求总量（QPS / RPM）
- 各接口请求量（按 URI/方法）
- 响应耗时（P50/P90/P95/P99）
- 错误率（4xx、5xx）
- 慢请求数量（超过阈值）

### 4.4 中间件与依赖（建议）
- 数据库连接池：活跃连接、空闲连接、等待连接
- SQL 执行耗时分布（可选）
- 缓存命中率（若使用 Redis）

## 5. 功能需求

### 5.1 指标采集（P0）
- 自动采集 Java 应用与主机关键指标
- 采集间隔可配置（默认 15s）
- 采集失败需记录日志并可观测

### 5.2 实时监控看板（P0）
- 提供统一监控看板（总览 + 细分页面）
- 支持按服务、实例、时间范围筛选
- 图表支持分钟级刷新（默认 10s~30s）

看板建议：
- 系统总览：CPU、内存、错误率、请求量
- JVM 专项：Heap、GC、线程
- 接口性能：TOP 慢接口、错误接口排行

### 5.3 告警机制（P0）
- 支持静态阈值告警
- 支持连续触发策略（如连续 3 个周期超限才报警）
- 支持告警恢复通知
- 告警渠道：邮件/企业微信/钉钉（至少一种）

### 5.4 日志与指标联动（P1）
- 告警触发后快速跳转对应时间段日志
- 支持 traceId 维度检索（若已接入链路追踪）

### 5.5 历史趋势分析（P1）
- 保留历史监控数据，支持天/周趋势分析
- 支持导出监控报表（CSV/图片）

## 6. 非功能需求

### 6.1 性能
- 监控采集对应用性能影响 < 5%
- 看板查询响应时间：95% 请求 < 2s

### 6.2 可用性
- 监控系统可用性目标 99.9%
- 采集端短暂异常具备重试机制

### 6.3 安全
- 监控页面需登录访问
- 指标接口需鉴权与权限控制
- 脱敏展示敏感信息（如数据库连接串）

### 6.4 可扩展性
- 支持新增自定义业务指标
- 支持多实例横向扩展

## 7. 推荐技术方案

建议采用：
- 应用埋点：Spring Boot Actuator + Micrometer
- 指标存储：Prometheus
- 可视化：Grafana
- 告警：Alertmanager

采集链路：
1. Java 后端暴露 `/actuator/prometheus`
2. Prometheus 定时拉取指标
3. Grafana 读取 Prometheus 展示图表
4. Alertmanager 根据规则触发告警

## 8. 告警规则建议

| 告警项 | 条件 | 持续时间 | 严重级别 |
|---|---|---|---|
| CPU 过高 | CPU > 80% | 5 分钟 | 高 |
| 内存过高 | 内存 > 85% | 5 分钟 | 高 |
| Full GC 频繁 | Full GC 次数 > 阈值 | 10 分钟 | 高 |
| 错误率升高 | 5xx 比例 > 5% | 3 分钟 | 高 |
| 响应变慢 | P95 > 2s | 5 分钟 | 中 |
| 请求突增 | QPS 超过基线 200% | 3 分钟 | 中 |

## 9. 验收标准

1. 监控页面可实时看到 CPU、内存、JVM、访问量、错误率。
2. 指标刷新延迟不超过 30 秒。
3. 阈值告警可触发并收到通知。
4. 可查看近 7 天历史趋势。
5. 监控接入不影响主流程功能与稳定性。

## 10. 实施计划

### P0（1~2 周）
- 接入 Actuator + Micrometer
- 部署 Prometheus + Grafana
- 完成基础 Dashboard 与 4 条核心告警

### P1（2~4 周）
- 增加 JVM 深度指标与慢接口分析
- 告警分级与恢复通知
- 增加历史趋势分析与导出

### P2（4~6 周）
- 日志联动与 trace 关联
- 自定义业务指标体系
- 多环境统一监控视图

## 11. 风险与应对

| 风险 | 说明 | 应对措施 |
|---|---|---|
| 指标过多导致存储压力 | 高基数标签导致 Prometheus 膨胀 | 规范标签维度，控制基数 |
| 告警误报较多 | 阈值设置不合理 | 增加持续时间与组合条件 |
| 监控本身不可用 | 监控组件单点故障 | 后续进行高可用部署 |
| 安全风险 | 指标泄露系统信息 | 鉴权、网络隔离、敏感信息脱敏 |
